// FaceApp — System Documentation

Laravel 13 React 19 Docker SQLite

FaceApp is a face recognition access control platform. It bridges a React-based enrollment kiosk, a Laravel REST API, and physical biometric access devices through a Java gateway service. Operators manage users and devices via a web admin panel; visitors enroll their face through the kiosk; the gateway syncs face data to hardware and handles door control.

The system is designed to operate fully on-premise with a reverse-proxied cloud API layer. All services run as Docker containers and communicate over an internal Docker network, with only HTTPS (443), HTTP (80), and device TCP ports (10010, 10011) exposed externally.

Enrollment Kiosk

React SPA with camera capture. Users self-enroll; face data is sent to the API and synced to hardware.

API Backend

Laravel 13 REST API. Manages users, devices, enrollments, and gateway communication.

Gateway SDK

Legacy Java service that translates API calls to the proprietary device TCP protocol.

Admin Panel

Laravel-rendered dashboard. Manage devices, users, and system settings.

Data Flow — Face Enrollment

# User completes enrollment on the kiosk
Browser (Camera)
  → POST /api/enrollments          # base64 photo + user metadata
    → FaceEnrollmentService
      → store photo to disk
      → GatewaySdkClient.mergeFace()   # for each active device
        → Java Gateway HTTP :8190
          → TCP to physical device :10010
      → GatewaySdkClient.findFace()    # verify registration
      → persist status to DB
  ← enrollment status JSON

Data Flow — Device Callbacks

# Device sends events back to the API (HTTP webhooks)
Physical Device
  → POST /api/device/callbacks/heartbeat         # every 60s
  → POST /api/device/callbacks/records           # on access event
  → POST /api/device/callbacks/person-registrations

03 Quick Start

The entire stack runs via Docker Compose. Before starting, ensure Docker and Docker Compose v2 are installed on the host.

Firewall: Open ports 80, 443, 10010 (device TCP), and 10011 (gateway SDK) on your VPS before proceeding.

1

Clone the repository and copy the environment file

git clone <repo-url> faceapp-1
cd faceapp-1
cp .env.example .env
2

Edit .env with your values

At minimum, set your domain names, a fresh Laravel app key, and the gateway device credentials. See the Environment Variables section for all required values.

FACEAPP_DOMAIN=faceapp.yourdomain.com
FACEAPP_API_DOMAIN=faceapp-api.yourdomain.com
LARAVEL_APP_KEY=base64:<run php artisan key:generate --show>
GATEWAY_DEVICE_KEY=your_device_key
GATEWAY_SECRET=your_device_secret
3

Point DNS to your VPS public IP

Create A records for both FACEAPP_DOMAIN and FACEAPP_API_DOMAIN pointing to the VPS IP. Caddy will automatically provision TLS certificates via ACME once DNS propagates.

4

Build and start all services

docker compose up -d --build

This builds the Laravel image (runs migrations via entrypoint), starts the Java gateway, Caddy proxy, and serves the React build.

5

Verify services are running

docker compose ps
docker compose logs -f faceapp-api

The health check endpoint GET /api/external/health (with a Bearer token) can be used to confirm the API is responding.

Local Development (Laravel)

cd faceapp-api
composer install
cp .env.example .env
php artisan key:generate
php artisan migrate

# Start all dev processes (server, queue, log tail, Vite)
composer run dev

06 API Routes

All API routes are prefixed with /api. Requests should target FACEAPP_API_DOMAIN.

Enrollment

MethodPathDescription
POST /api/enrollments Create a new face enrollment. Accepts base64 photo + user metadata. Triggers sync across all active devices. Returns enrollment object with status.
GET /api/enrollments/{public_id} Retrieve enrollment status by public UUID. Used to poll enrollment progress.

Frontend Dashboard Data

MethodPathDescription
GET /api/app/dashboard Returns users, devices, and summary data for the React dashboard view.

Device Status

MethodPathDescription
GET /api/device/status Returns current status of all registered devices.

Device Callbacks (Webhooks from Hardware)

These endpoints are called by the physical device, not by your application. No authentication required — the device posts raw JSON payloads.

MethodPathDescription
POST /api/device/callbacks/heartbeat Device heartbeat. Updates last_seen_at, person/face counts, disk space.
POST /api/device/callbacks/records Access event (door open/denied). Logged to device_events.
POST /api/device/callbacks/person-registrations Device-initiated user registration (e.g. card scan on device). Triggers user import flow.

External Integration API

Authenticated with Authorization: Bearer {FACEAPP_EXTERNAL_API_TOKEN}. Token is validated with hash_equals() to prevent timing attacks.

MethodPathAuthDescription
GET /api/external/health Bearer Health check. Returns 200 OK if the API is reachable and token is valid.
POST /api/external/open-gate Bearer Triggers door/turnstile open command on a specified device. Used by qparking after a paid exit event.

08 Key Services

Business logic lives in faceapp-api/app/Services/. Services are injected via Laravel's container.

FaceEnrollmentService

Orchestrates the complete enrollment workflow. Extracts the base64 photo, stores it to disk, iterates over all active devices, calls mergeFace() on the gateway, then calls findFace() to verify registration. Tracks per-device status and writes back to the enrollments table. Configurable retry count and delay.

GatewaySdkClient

HTTP client for the Java gateway service at http://gateway:8190/api. Wraps all device, person, and face operations. Supports per-device credentials with a fallback to global config. Handles auth failure retries transparently. Key methods: upsertPerson(), mergeFace(), findFace(), output() (open door), reboot(), setServerConfig().

ManagedUserSyncService

Handles all user-to-device sync operations. syncUserToDevice() upserts a user's person record on a specific device. deleteUserAcrossActiveDevices() removes a user from all hardware. importUsersFromDevice() pulls an existing user list from hardware and creates managed_users records. Updates managed_user_device_syncs with per-device results.

DeviceMonitoringService

Processes inbound webhooks from hardware. recordHeartbeat() updates device online status, person count, face count, and free disk space. recordAccessRecord() logs door access events. recordPersonRegistration() handles device-initiated registrations. All events are appended to device_events for audit.

SystemSettingsService

Singleton configuration provider. Reads gateway URLs, callback URLs, retry counts, and timing windows from the system_settings table, with environment variable fallbacks. Consumed by GatewaySdkClient and FaceEnrollmentService. Editable at runtime via /admin/settings without redeploying.

09 External Integrations

Java Gateway SDK Service (Internal)

A legacy Java 8 binary bundled in the gateway-sdk-service_v2.0.6_20240117/ directory. It runs in Docker, listens on three ports, and translates REST calls from Laravel into the proprietary device protocol.

PortPurpose
8190 (internal)REST API consumed by Laravel — all device/person/face operations
10010 (external)TCP socket the physical device connects to
10011 (external)SDK secondary port

Image URL constraint: Physical devices fetch face images over plain HTTP (not HTTPS). GATEWAY_IMAGE_BASE_URL must be a plain http:// URL. The application stores photos at both a public HTTPS URL (for the app) and the HTTP URL (for devices).

qparking-local (External)

An external parking management system that calls POST /api/external/open-gate to release a turnstile or gate after a vehicle exit is processed. The integration is authenticated with a shared bearer token set in FACEAPP_EXTERNAL_API_TOKEN.

# qparking calls this after a paid exit event
POST /api/external/open-gate
Authorization: Bearer {FACEAPP_EXTERNAL_API_TOKEN}
Content-Type: application/json

{
  "device_key": "GATE-001"   // optional; omit for default device
}

Physical Face Recognition Device

Hardware biometric device from a Chinese OEM vendor. It operates as follows:

→

Outbound (Device → System)

Connects to gateway TCP on port 10010. Sends heartbeat every 60s, access records on recognition events, and person registration events when a card is tapped on the device.

←

Inbound (System → Device)

Laravel triggers commands via the Java gateway: add/update/delete persons, enroll faces, open door, reboot, set callback server URL. All routed through GatewaySdkClient.
